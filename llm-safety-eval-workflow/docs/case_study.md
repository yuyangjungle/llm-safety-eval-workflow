# Case Study: LLM 安全评测数据生产与质量验收 Workflow

## 背景

大模型安全评测往往不是单纯“问几个危险问题”，而是需要把抽象安全风险拆成可生产、可验收、可迭代的数据资产。本项目围绕 AI 数据与安全中台类岗位能力，搭建一个离线 MVP，验证从数据策略到质量验收再到模型评测反馈的完整闭环。

## 问题定义

目标是回答三个问题：

1. 如何把 LLM 安全问题拆成稳定的数据分类体系？
2. 如何让评测样本具备可生产、可校验、可复盘的结构？
3. 如何根据模型失败样本反推下一轮数据补充策略？

## 解决方案

项目将流程拆为五层：

| 层级 | 产物 | 说明 |
| --- | --- | --- |
| 数据策略 | `data/risk_taxonomy.json` | 定义 8 类安全风险、预期行为和评估关注点 |
| 样本标准 | `docs/schema.md` | 定义 prompt、标签、难度、严重度、rubric 等字段 |
| 数据生产 | `scripts/generate_samples.py` | 从种子样本扩展到模板化合成样本 |
| 质量验收 | `scripts/evaluate_quality.py` | 统计覆盖率、完整率、去重率、难度分布 |
| 模型评测 | `scripts/judge_outputs.py` | 对模型输出进行 rubric judge、bad case 归因和数据飞轮建议 |

## 当前结果

- 8 类风险分类体系：隐私泄露、提示注入、有害行为、幻觉事实性、偏见公平性、不安全工具调用、数据合规、多模态上下文错配。
- 32 条 MVP 样本：8 条人工种子样本 + 24 条模板化合成样本。
- 质量门禁：Schema 完整率、风险覆盖率、Prompt 去重率、Rubric 完整率均为 100%。
- 模型评测样例：对 `baseline_naive_v0` 与 `safety_workflow_v1` 两个离线候选输出进行 rubric judge，输出 bad case 和补样建议。

## 设计取舍

- 当前版本优先验证数据产品链路，而不是追求大规模样本量。
- 合成样本采用模板化扩展，保证可解释和可复盘，后续可替换为真实 LLM 生成。
- Judge 采用离线 rubric 规则，便于本地复现；后续可接入 LLM-as-judge API 和人工抽检一致性评估。

## 面试可讲点

- 如何把“安全”这种抽象问题变成 taxonomy、schema 和质量门禁。
- 为什么样本需要包含 expected behavior 和 judge rubric，而不是只有 prompt。
- 如何通过 bad case 分析反推数据策略，比如某类风险样本不足、rubric 不可判定、prompt 难度过低。
- 如何从 MVP 演进到真实产线：接入真实模型输出、LLM-as-judge、人工抽检、自动补样和版本管理。

## 边界说明

当前项目是公开展示版 MVP，不声称服务真实业务或接入 Seed 模型。简历中应表述为“搭建离线 MVP / 设计 Workflow / 输出质量报告和评测样例”，不要写成“上线生产系统”。
